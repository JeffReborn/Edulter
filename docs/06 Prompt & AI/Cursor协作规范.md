## 一、文档目标

本文件用于定义在教育咨询顾问助手 Demo 开发过程中，如何让 Cursor 参与代码实现、重构、修复与推进，确保：

- 开发节奏可控
- 任务边界清晰
- 上下文不漂移
- 架构不被随意破坏
- 每次改动都可验证、可回退、可复盘

本文件不是技术架构文档，也不是产品文档，而是：

**你与 Cursor 协同开发时的执行规则。**

## 二、Cursor 的定位

在当前项目中，Cursor 的角色定位为：

- 高执行力工程实现助手
- 多文件代码修改助手
- 基于现有文档与代码的实现者
- 小步推进的开发协作者

Cursor **不是**：

- 自动替你做产品决策的 CTO
- 自动决定范围边界的产品经理
- 不受约束的全自动整项目生成器
- 可以脱离文档和规则随意大改的代理人

### 当前协作分工建议

#### 你 / ChatGPT 负责
- 定义产品边界
- 定义 MVP 范围
- 定义系统架构
- 定义数据模型
- 定义 API 合同
- 定义输出 Schema
- 定义任务拆分与优先级
- 审核 Cursor 的产出是否偏题

#### Cursor 负责
- 根据明确任务实现代码
- 修改多文件
- 搭建模块骨架
- 修复类型错误
- 接入接口
- 写基础校验与结构
- 补充局部测试与错误处理

## 三、当前协作核心原则

### 1. Cursor 只执行“已定义任务”
不要把模糊的产品愿望直接丢给 Cursor。

错误示例：
- 帮我做一个完整商用 AI SaaS
- 把整个产品从前端到后端都写完
- 帮我自由发挥把系统补完整

正确示例：
- 读取指定文档后，只实现文档上传模块的最小闭环
- 只实现 `POST /api/qa/ask`
- 只补前端上传表单与状态展示
- 只根据 `数据模型.md` 生成 Prisma schema

### 2. 一次只做一个明确闭环
每次让 Cursor 聚焦一个模块、一个接口、一个页面或一个局部重构。

不要一次同时做：
- 数据模型
- 上传模块
- 问答模块
- 画像模块
- 跟进消息
- 客户详情页

### 3. 先读文档，再写代码
Cursor 在开始实现前，应先读取与你当前任务直接相关的文档，而不是凭空猜测。

### 4. 不允许 Cursor 自行扩大范围
如果任务是：
- 实现上传接口

那么 Cursor 不应顺手去：
- 重写数据模型
- 改整个 UI 风格
- 增加权限系统
- 顺便把问答和画像也一起做了

### 5. 所有关键改动都必须可解释
Cursor 输出后，你应能清楚知道：
- 改了哪些文件
- 为什么这样改
- 哪些是本任务范围内的
- 哪些风险仍未处理

### 6. 每完成一个稳定步骤就提交一次 commit
不要连续让 Cursor 堆 10 个大改动后才保存。  
应该小步完成、小步验证、小步提交。

## 五、推荐的 Cursor 任务输入格式

每次给 Cursor 下任务时，建议遵循下面的结构。

### 任务输入模板

#### 1. 指定必须先读的文档
例如：
- 先阅读 `系统架构.md`、`数据模型.md`、`API合同.md`、`文档上传模块.md`

#### 2. 明确本次只做什么
例如：
- 本次只实现文档上传接口
- 本次只实现 Prisma schema
- 本次只实现问答模块前端页面
- 本次只实现客户详情页读取逻辑

#### 3. 明确本次不要做什么
例如：
- 不要修改 auth
- 不要改 UI 风格系统
- 不要顺便实现别的模块
- 不要修改未提及的数据表结构

#### 4. 明确验收标准
例如：
- 接口能通过 typecheck
- 前端可发起请求并看到状态
- 至少支持一种文件格式
- 返回字段必须符合 API 合同

#### 5. 明确输出要求
例如：
- 列出修改了哪些文件
- 说明还存在哪些未解决问题
- 运行 typecheck / lint 后汇报结果

## 九、当前项目最重要的协作限制

### 1. 不允许擅自改数据模型
如果本次任务不是数据模型调整，就不应让 Cursor随意修改 Prisma schema。

### 2. 不允许擅自扩大模块范围
例如：
让 Cursor 做问答模块时，不应顺手把客户记录页也大改。

### 3. 不允许把 Prompt 逻辑散落到各处
Prompt 和 AI 结构相关逻辑应尽量集中在：
- `lib/ai`
- `server/services`
而不是散落在页面组件里。

### 4. 不允许页面直接碰数据库
前端页面只能通过 API 或明确的 action 层访问后端能力。

### 5. 不允许把 route 写成超厚业务层
Route Handler 应尽量轻：
- 接收请求
- 校验参数
- 调用 service
- 返回响应

核心业务逻辑应放在 service 层。

## 十一、推荐的每轮执行流程

### 第一步：明确任务
先在 Obsidian 里确认：
- 本次任务名
- 依赖文档
- 验收标准
- 非目标范围

### 第二步：给 Cursor 任务
让 Cursor 先读文档，再只做当前任务。

### 第三步：检查返回
检查：
- 改了哪些文件
- 是否偏题
- 是否越界
- 是否真的符合 API 合同和 Schema

### 第四步：本地验证
至少验证：
- typecheck
- lint
- 基础运行
- 接口字段是否正确

### 第五步：提交与记录
如果结果稳定：
- Git commit
- 如有关键决定，更新对应文档
- 如有问题，记录到 [[阻塞问题]]

## 十五、当前常见协作错误提醒

### 1. 给 Cursor 任务太大
会导致它：
- 漏细节
- 改太多
- 偏离原始目标

### 2. 不先给文档上下文
会导致它依赖猜测，而不是依赖项目基线。

### 3. 不限制“不要做什么”
会导致它顺手扩大范围。

### 4. 不做结果校验
会导致你以为“代码写出来了”，但实际字段不稳定、结构不对、架构偏了。

### 5. 不做小步 commit
会导致一旦出问题，很难回退。
