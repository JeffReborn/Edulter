## 一、文档目标

本文件用于统一管理教育咨询顾问助手 Demo 的整体开发任务，作为当前项目的主任务池和执行总入口。

它的作用是：

- 汇总所有待办任务
- 明确优先级
- 控制开发顺序
- 防止范围膨胀
- 作为后续 `当前Sprint`、`本周任务`、`验收清单` 的上游任务来源

---

## 二、使用原则

### 1. 这里只放“项目级任务”
本页记录的是跨模块、跨阶段、可执行的项目任务，不写太碎的临时灵感。

### 2. 任务必须有优先级
当前统一使用：

- `P0`：不做就无法形成 Demo 闭环
- `P1`：不是第一优先，但对产品完成度明显有帮助
- `P2`：未来可做，不进入当前主开发节奏

### 3. 任务要尽量可验收
每条任务最好能回答：
- 做完后，项目会多出什么能力？
- 如何判断它完成了？

### 4. 任务总表不是 Sprint 看板
这里是“总池”，不代表本周全部都做。  
每周实际做哪些，应下沉到：
- [[当前Sprint]]
- [[本周任务]]

## 八、建议的当前最小闭环定义

当以下链路全部成立时，可认为 Demo 核心闭环完成：

1. 用户可上传内部资料
2. 系统可处理文档并入库
3. 用户可基于资料发起问答并看到出处
4. 用户可粘贴咨询文本提取客户画像
5. 用户可基于画像生成跟进消息
6. 系统可保存客户主体、咨询记录、画像、跟进消息
7. 用户可在客户记录中查看最新结果

再做 AI 主闭环：
- 文档上传
- 知识问答

### 第三步
最后补客户记录承载与页面体验

不要一开始先做：
- 客户列表花哨 UI
- 历史时间轴
- 多筛选条件
- 管理后台

### 拆分原则
- 一个任务卡尽量只做一个模块能力
- 一次只让 Cursor 改一个相对独立的闭环
- 每完成一组任务就提交一次 commit
- 不要让 Cursor 在一轮里同时大改前端、后端、数据模型和 Prompt
