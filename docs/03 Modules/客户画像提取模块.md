## 一、模块目标

实现从客户咨询内容中提取结构化客户画像的能力，使教育咨询顾问能够快速把非结构化沟通内容整理为可用于后续跟进和判断的业务信息。

该模块的核心目标是把：

- 咨询对话
- 沟通纪要
- 顾问整理后的客户信息

转换为：

- 结构化客户画像
- 当前阶段判断
- 主要关注点
- 风险点

并将结果保存到客户记录中，供后续跟进消息生成和客户回看使用。

---

## 二、模块定位

该模块的职责是：

- 接收标准化后的咨询文本
- 提取客户核心信息
- 输出结构化画像
- 创建或更新客户主体
- 保存原始文本记录
- 保存画像结果

该模块 **不负责**：

- 直接处理原始音频文件
- 直接处理聊天截图 OCR
- 自动发送跟进消息
- 完整 CRM 客户管理
- 多轮跟进时间轴管理

## 七点五、多轮咨询下的画像更新说明

### 1. 当前业务判断
客户画像不应被理解为“一次提取后永久固定的结果”。

在真实教育咨询场景中，客户信息会随着后续沟通不断补充和变化，例如：

- 首咨时只知道学生阶段和目标国家
- 二咨后补充预算、城市偏好、语言基础
- 三咨后补充家长真实顾虑、申请时间线、学校层级倾向

因此，客户画像本质上应被理解为：

**系统基于当前已知咨询内容，对客户状态做出的阶段性结构化总结。**

### 2. 为什么画像必须支持更新
如果后续有新的咨询内容，但画像仍停留在首咨版本，会导致：

1. 客户记录逐渐失真
2. 跟进消息与当前客户情况脱节
3. 顾问无法判断客户状态是否发生变化
4. 系统只能作为“一次性首咨工具”，而不能融入真实业务流程

因此，本模块应支持：

**随着新增咨询文本不断重新提取和更新客户画像。**

### 3. Demo 阶段的实现策略
为了控制复杂度，Demo 第一版不做复杂的自动多轮汇总系统，但应支持最小可行的“手动更新画像”能力。

#### 顾问在某个客户详情页中可以：
1. 粘贴新一轮咨询文本
2. 再次触发画像提取
3. 基于最新画像继续生成跟进消息

系统应完成：
1. 新增一条 `conversation_record`
2. 新增一条新的 `client_profile`
3. 更新 `clients` 中的高频摘要字段
4. 保留旧画像，不直接覆盖历史画像记录

### 4. 当前推荐的数据处理方式
为了兼顾“当前可用”和“历史可回看”，推荐采用以下模式：

#### `clients`
保存：
- 当前客户的最新摘要状态
- 例如学生阶段、目标国家、预算范围、当前阶段等高频字段

#### `client_profiles`
保存：
- 每次画像提取生成的一条版本记录
- 对应当时的咨询输入和模型输出

#### `conversation_records`
保存：
- 每一轮新增咨询文本
- 作为画像提取的原始输入依据

这样可以实现：

- 页面默认看“当前最新状态”
- 数据层仍保留“历史演化过程”

### 5. 当前页面层面的收敛方式
虽然底层应支持多轮更新，但 Demo 阶段前端不需要做复杂版本管理界面。

当前页面只需优先展示：
- 最新画像结果
- 最新咨询文本
- 当前阶段判断
- 当前主要关注点和风险点

后续版本再考虑增加：
- 历史画像版本列表
- 首咨与二咨画像差异对比
- 画像变化轨迹展示

### 6. 与跟进消息生成的关系
画像更新的直接业务意义在于：

**跟进消息不应基于过期画像生成。**

每当客户有新的沟通内容并完成新画像提取后，系统应允许顾问基于“最新画像”重新生成新一轮跟进消息。

也就是说：

- 新咨询内容
- 新画像
- 新跟进消息

这三者应形成连续链路。

### 7. 当前模块边界补充
本模块支持“围绕同一客户持续更新画像”，但当前仍然不做：

- 自动合并所有历史咨询为统一长上下文
- 自动判断哪一轮信息应覆盖旧信息
- 复杂画像版本对比页面
- 多顾问协同修订画像
- 人工审核流

这些能力可以进入后续 MVP / V1。

### 8. 当前产品意义
加入这层“多轮画像更新”逻辑后，本模块就不再只是首咨后的静态提取工具，而更接近真实业务场景下的动态客户理解引擎：

- 每轮沟通都能补充客户理解
- 每次提取都能更新当前判断
- 后续跟进动作能跟着客户状态同步变化

这会显著提升产品的长期可用性和业务真实性。

## 十二、技术要点

### 1. 输入统一为文本
当前模块只处理标准化后的文本输入，不处理音频和图片本体。

这是为了保证模块职责清晰，避免把“输入预处理”和“画像提取”混为一体。

### 2. Prompt 需要强约束
Prompt 应明确要求模型：

- 按固定字段输出
- 信息不足时允许留空或标记不确定
- 不要编造不存在的信息
- 输出尽量面向顾问业务使用

### 3. 输出必须过 schema 校验
模型输出不能直接返回前端，必须经过后端结构校验。

如果 schema 不合法，应返回错误或触发兜底逻辑。

### 4. 高价值字段应拆出保存
例如：

- `studentStage`
- `targetCountry`
- `budgetRange`
- `currentStage`

这些字段应作为独立字段保存，而不是只塞进 JSON。

### 5. structured_json 仍需保留
完整画像结构应整体保存，方便后续升级字段或重新利用历史数据。

### 6. 客户主体与画像要分离
不要把画像直接只写进 `clients` 表里。  
正确做法是：

- `clients` 作为客户主体
- `client_profiles` 作为画像版本记录

这样后续才能保留历史提取结果。

## 十三、输出策略建议

### 当前输出策略
优先追求：

1. 结构清晰
2. 信息可用
3. 业务可读
4. 不乱编造

### 当信息不足时
系统应允许：

- 某些字段为空
- 某些字段标记为待确认
- 风险点里指出“信息仍不完整”

### 不应追求
- 表面看起来很完整但实际凭空补全的信息
- 冗长散文化总结
- 没有业务价值的漂亮措辞

## 十四、异常处理

### 1. 输入为空
用户提示：
- 请先粘贴咨询内容后再提取画像

### 2. 输入文本过短
用户提示：
- 当前内容不足以提取完整画像，请补充更多咨询信息

### 3. 模型提取失败
用户提示：
- 当前画像提取失败，请稍后重试

### 4. 输出结构不合法
后端应返回：
- 结构化提取失败，请重试或检查输入内容

### 5. 信息明显不足
用户提示：
- 当前画像仅提取到部分信息，仍有关键字段待补充

满足以下条件时，可认为该模块达标：

1. 用户可以粘贴咨询文本并成功发起提取
2. 系统能返回结构化客户画像

## 十六、风险点

### 1. 输入质量波动大
如果顾问输入的文本过短、过乱或信息缺失，画像质量会明显下降。

### 2. 模型容易“脑补”
如果 Prompt 约束不够，模型可能会补出看似完整但实际不存在的信息。

### 3. 字段设计过多会增加不稳定性
首版字段不宜过于复杂，先保证高价值字段稳定输出。

### 4. 容易把输入预处理职责混进来
音频转写和 OCR 很有价值，但不应在当前模块中直接承担。

### 5. 容易滑向完整 CRM
本模块只负责“提取和沉淀客户画像”，不负责完整客户管理体系。
