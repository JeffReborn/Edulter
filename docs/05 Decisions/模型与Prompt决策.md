## 一、文档目标

本文件用于记录教育咨询顾问助手 Demo 中，与模型选择、Prompt 设计、输出约束、AI 调用策略相关的关键决策，避免后续出现：

- Prompt 目标反复变化
- 模型职责混乱
- 输出风格不稳定
- 后端、前端、Prompt 三方字段不一致
- 调优过程没有可追溯依据

它的作用是：

- 固化当前 AI 层的核心判断
- 解释为什么这样设计 Prompt
- 说明模型调用应服务什么目标
- 为后续 Prompt 迭代与质量优化提供基线

---

## 二、使用原则

### 1. 这里只记录“关键 AI 决策”
适合写在这里的内容包括：

- 某类任务使用什么模型能力
- Prompt 应优先追求什么目标
- 输出结构如何约束
- 哪些 AI 能力当前不做
- 多轮更新的 AI 处理原则是什么

### 2. 每条决策都应记录原因
不仅写“决定了什么”，还应写：
- 为什么这样选
- 放弃了哪些路线
- 当前阶段这样做有什么好处

### 3. 当前以 Demo 闭环优先
所有模型与 Prompt 决策，优先服务：
- 输出稳定
- 结构清晰
- 易于联调
- 便于顾问实际使用

而不是追求“最聪明”或“最炫技”的 AI 表现。

## 决策 01：知识问答、客户画像提取、跟进消息生成应视为三种不同 AI 任务

### 状态
`Decided`

### 决策内容
当前 AI 能力明确拆分为三类任务：

1. 知识问答
2. 客户画像提取
3. 跟进消息生成

三者虽然都调用 LLM，但不应共用同一个“万能 Prompt”或同一种输出逻辑。

### 决策原因
这三类任务的目标完全不同：

- 知识问答：强调基于资料、带出处、避免幻觉
- 客户画像提取：强调结构化提取、字段稳定、避免编造
- 跟进消息生成：强调表达自然、贴近业务、便于发送

如果混在一起，会导致：
- Prompt 目标冲突
- 输出结构混乱
- 调优方向不清晰

### 当前影响
应为三类任务分别设计：
- Prompt
- 输出约束
- service 逻辑
- 校验方式

## 决策 02：知识问答 Prompt 的首要目标是“基于资料回答”，不是“回答得像聊天机器人”

### 状态
`Decided`

### 决策内容
知识问答 Prompt 必须优先约束模型：

- 仅依据提供的上下文回答
- 没有依据时明确说信息不足
- 尽量返回可读、简洁、业务化的回答
- 不自由扩写企业资料之外的内容

### 决策原因
知识问答模块的产品价值在于：

**基于企业资料给出有依据的回答。**

如果 Prompt 鼓励模型自由发挥，结果会更像泛 AI 问答，而不是企业知识助手。

### 当前影响
Prompt 中必须明确：
- 优先使用上下文
- 不要凭空补充政策、服务承诺或细节
- 当依据不足时允许保守回答

## 决策 03：客户画像提取 Prompt 的首要目标是“结构稳定”，不是“总结写得漂亮”

### 状态
`Decided`

### 决策内容
客户画像提取 Prompt 的首要目标应是：

- 按固定字段提取
- 字段含义一致
- 信息不足时不乱补
- 输出可直接落库和驱动后续模块

而不是生成一段“看起来很会总结”的自然语言分析。

### 决策原因
客户画像模块的核心价值在于：

**把咨询文本变成结构化业务对象。**

如果输出只是漂亮总结，会带来：
- 前端难以渲染
- 数据库难以落库
- 跟进消息模块难以复用
- 后续联调成本很高

### 当前影响
Prompt 应明确要求：
- 按字段输出
- 不乱编
- 高价值字段优先
- `mainConcerns`、`riskFlags` 等字段必须稳定为数组

## 决策 04：跟进消息 Prompt 的首要目标是“可发送、可微调、自然”，不是“华丽写作”

### 状态
`Decided`

### 决策内容
跟进消息生成 Prompt 应优先追求：

- 可直接作为顾问对外消息草稿
- 自然
- 清晰
- 不夸张
- 不模板腔
- 不过度销售化

### 决策原因
这个模块生成的内容最终要给顾问发给客户。  
如果写得太“AI”、太“官”、太“销售”，即使语言流畅，也没有实际使用价值。

### 当前影响
Prompt 中应明确要求：
- 贴近教育咨询顾问真实表达
- 不要写空泛安抚句
- 不要过度承诺
- 不要编造服务能力
- 不要输出“以下是为您生成的话术”这类包装句

## 决策 05：客户画像首版字段要少而稳，不要一次性追求过多字段

### 状态
`Decided`

### 决策内容
客户画像首版应先聚焦高价值且较稳定的字段，例如：

- `studentStage`
- `targetCountry`
- `targetProgram`
- `budgetRange`
- `timeline`
- `englishLevel`
- `parentGoals`
- `mainConcerns`
- `riskFlags`
- `currentStage`

### 决策原因
字段越多，Prompt 越复杂，模型输出越不稳定。  
首版如果一次性加入太多边缘字段，反而会降低整体质量。

### 当前影响
当前 Prompt 设计和 schema 校验应围绕“少而稳”的字段集合构建。  
更细字段可先放到：
- `structuredJson`
- 后续版本扩展池

## 决策 06：客户画像允许“信息不足”，但不允许“编造完整”

### 状态
`Decided`

### 决策内容
当咨询文本中信息不足时，模型应允许：

- 留空字段
- 返回空数组
- 使用“待确认”类保守值
- 在 `riskFlags` 中指出信息不完整

但不允许为了让结果“看起来完整”而捏造：
- 预算
- 时间线
- 学生背景
- 英语水平
- 家长明确诉求

### 决策原因
在教育咨询场景里，假信息比缺信息更危险。  
如果模型编造信息，后续：
- 跟进消息会偏
- 客户记录会失真
- 顾问会逐渐不信任系统

### 当前影响
Prompt 和后端校验都应接受“部分字段为空”的情况。  
不要为了追求“高完成度”而逼模型乱补。

## 决策 07：跟进消息首版只做两种风格，并要求差异明确

### 状态
`Decided`

### 决策内容
当前只先支持两种风格：

- `wechat_short`
- `semi_formal`

并要求两种风格必须有真实差异，不只是同一句话改几个词。

### 决策原因
过早支持过多风格会导致：
- Prompt 复杂度上升
- 风格区分不明显
- 联调成本增大

先把两种风格做清楚，比一开始铺很多风格更实用。

### 当前影响
Prompt 中应明确区分：
- 长度
- 语气
- 信息完整度
- 使用场景

## 决策 08：所有核心 AI 输出都必须服从 `输出Schema定义`

### 状态
`Decided`

### 决策内容
后续所有 Prompt 设计，都必须以 [[输出Schema定义]] 为上游约束，而不是自由定义字段。

### 决策原因
如果 Prompt 自己定义一套字段、后端又定义一套、前端再假设一套，最终会导致：

- 字段名混乱
- 联调困难
- Cursor 反复返工
- 数据难以落库

### 当前影响
以下文档必须对齐：
- [[知识问答Prompt]]
- [[客户画像提取Prompt]]
- [[跟进消息生成Prompt]]

## 决策 09：Prompt 中要明确“模型不知道时可以保守回答”

### 状态
`Decided`

### 决策内容
当前 Prompt 设计必须允许模型在依据不足时：

- 承认不确定
- 返回信息不足
- 只输出当前可确认部分
- 不强行填满所有字段

### 决策原因
很多模型一旦被要求“尽量完整”，就会倾向于脑补。  
当前阶段我们更需要的是：
- 稳定
- 保守
- 可控

而不是表面完整但充满隐性错误。

### 当前影响
这条原则应体现在三类 Prompt 中，尤其是：
- 知识问答
- 客户画像提取

## 决策 10：Prompt 逻辑必须集中管理，不允许散落在页面和 route 中

### 状态
`Decided`

### 决策内容
Prompt 相关逻辑应尽量集中到统一位置，例如：
- `lib/ai`
- `server/services`

不允许把 Prompt 内容散落在：
- 页面组件
- 多个 route 文件
- 随手拼接的 util 文件

### 决策原因
Prompt 是当前产品的核心业务逻辑之一。  
如果散落到各处，会导致：
- 难以统一更新
- 难以记录版本
- 难以调优
- 未来拆分困难

### 当前影响
工程实现时应坚持：
- route 轻
- service 负责业务
- ai 层负责 Prompt 构造与模型调用

## 决策 11：首版不追求复杂 Prompt 编排和多模型路由

### 状态
`Decided`

### 决策内容
当前不做：
- 多模型动态路由
- 多阶段 Prompt pipeline
- agent 化编排
- 复杂自反思链路
- Prompt A/B 实验平台

### 决策原因
当前阶段最重要的是跑通：

- 可解释
- 可联调
- 可维护

的最小闭环，而不是做复杂 AI 系统工程。

### 当前影响
首版每条能力链路都尽量收敛为：

- 明确输入
- 单次核心 Prompt
- 输出校验
- 结构化返回

## 决策 12：多轮咨询下，画像和跟进消息应随“最新有效信息”更新

### 状态
`Decided`

### 决策内容
当前产品逻辑应默认：

- 新一轮咨询文本可生成新画像
- 新画像可驱动新跟进消息
- 最新页面优先展示最新结果
- 历史记录保留，但当前工作默认基于最新版本

### 决策原因
如果仍基于首咨画像生成后续跟进消息，内容很快会失真。  
因此 Prompt 层应默认服务“当前最新客户状态”。

### 当前影响
后续 Prompt 输入组装时，应优先读取：
- 最新 `conversation_record`
- 最新 `client_profile`

除非显式指定旧版本。

## 五、当前已确定的 Prompt 设计方向总结

当前三类 Prompt 的重点分别是：

### 知识问答 Prompt
重点：
- 基于资料
- 带出处
- 不乱编

### 客户画像提取 Prompt
重点：
- 结构稳定
- 字段统一
- 信息不足时保守

### 跟进消息生成 Prompt
重点：
- 可发送
- 自然
- 与客户情况相关
- 不模板腔

## 六、当前不建议反复摇摆的已定事项

以下内容当前已经确定，除非有强新证据，否则不建议反复推翻：

- 三类 AI 任务分别设计 Prompt
- 知识问答必须基于资料和出处
- 客户画像以结构稳定为优先
- 跟进消息以可发送和自然为优先
- 画像首版字段少而稳
- 信息不足时允许空缺，不允许脑补
- 跟进消息首版只做两种风格
- Prompt 必须服从 `输出Schema定义`
- Prompt 逻辑集中管理
- 当前不做复杂 Prompt 编排
- 多轮咨询下默认基于最新状态生成

## 八、当前处理原则

对于模型与 Prompt 层面的新增想法，当前统一按以下顺序判断：

1. 是否会让输出更稳定？
2. 是否会让前后端联调更容易？
3. 是否会减少 Prompt 漂移和字段混乱？
4. 是否会明显拖慢当前 Demo 交付？
5. 是否属于下一阶段优化，而不是当前必需？

如果某个想法只是“更炫”但不明显提升稳定性，应优先延后。
