## Deferred（暂缓决策）

以下事项当前仅记录问题与可选方向，**不在 Task Card 03 阶段做最终决策**。等进入对应模块任务卡（QA / Profile / Followup）后再定稿。

### 1) AI 客户端实现策略（Deferred）

- **问题**：后续 `qa/profile/followup` 的 AI 调用应采用什么实现策略？
- **背景**：目前仅建立 `src/lib/ai` 的最小接口与占位实现（不做真实调用），待后续任务卡接入。
- **需要决策的点**：
  - **模型选择**：是否统一一个模型/供应商，还是按模块（QA/Profile/Followup）分别选型？
  - **Prompt 管理**：Prompt / schema / 版本号应集中管理还是按模块拆分管理？
  - **结构化输出校验**：输出 schema 校验放在 ai wrapper 层、service 层，还是 API 层统一处理？
- **暂定做法（当前）**：保持 `AiClient` 接口稳定，后续在实现任务卡中替换 `aiClient` 为真实实现，不提前散落 prompt。
- **决策时机**：任务卡 07/08（QA）、任务卡 10/11（画像）、任务卡 13/14（跟进消息）。

### 2) Service 的实例化方式（Deferred）

- **问题**：在 API route / server action 中如何创建/复用 service 实例？
- **背景**：当前 service 层采用 `createXxxService(deps)` 工厂函数 + 依赖注入（db/ai），便于测试与扩展。
- **可选方向**：
  - **方向 A（简单直连）**：在每个 handler 内直接 `createXxxService({ db: prisma, ai: aiClient })`。
  - **方向 B（集中组合）**：新增 `src/server/services/index.ts`，统一组装并导出（例如 `services = { document, qa, ... }`），减少重复注入代码。
- **暂定做法（当前）**：先用方向 A，等模块 API 增多后再评估是否抽 `services/index.ts`。
- **决策时机**：开始批量落地 API route（任务卡 04 起）或 service 依赖注入出现明显重复时。
