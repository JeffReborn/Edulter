## 一、文档目标

本文件用于定义教育咨询顾问助手从当前 Demo 阶段到后续 MVP、V1 的演进路线，确保：

- 当前版本边界清晰
- 后续升级方向有序
- 不把未来版本的需求误塞进当前开发
- 产品演进有阶段感，而不是无限膨胀

本文件关注的是：

**这个产品下一步怎么演进，每个阶段大概解决什么问题。**

---

## 二、版本规划原则

### 1. 先验证价值，再做平台化
当前产品首先要验证的是：

- 知识问答是否有真实价值
- 客户画像提取是否可用
- 跟进消息生成是否能真正帮助顾问

在这些核心价值没有被验证前，不应过早投入大量精力去做平台化能力。

### 2. 先做主闭环，再做增强层
当前最优顺序应是：

1. 跑通主闭环
2. 增强结果稳定性
3. 增强回看与管理能力
4. 再逐步进入平台能力

### 3. 每一阶段都应有明确目标
每个版本的目标都应清楚回答：

- 当前阶段主要验证什么
- 当前阶段交付什么
- 当前阶段明确不做什么

### 4. 后续版本不是当前版本的“功能垃圾桶”
如果某个功能进入下一阶段，不代表它一定会做。  
它只是说明：

**当前不做，后续再评估。**

## 四、Stage 1：Demo v0.1

### 当前定位
可演示版本 / 价值验证版本 / 非正式商用版本

### 当前目标
验证三条核心主线是否成立：

1. 知识问答
2. 客户画像提取
3. 跟进消息生成

并建立最小可运行闭环：

1. 上传资料
2. 基于资料问答
3. 粘贴咨询文本
4. 提取客户画像
5. 生成跟进消息
6. 保存到客户记录中
7. 查看最新结果

### 当前应交付
- 文档上传最小闭环
- 问答最小闭环
- 客户画像提取最小闭环
- 跟进消息生成最小闭环
- 轻量客户记录列表与详情
- 基础 loading / error / empty state

### 当前不做
- 录音转写
- 聊天截图 OCR
- CRM 漏斗
- 自动触达
- 多租户
- 管理后台
- 复杂权限
- 高级 RAG 优化
- Prompt 配置后台

### 当前成功标准
- 可以完整演示一条真实业务闭环
- 输出结构稳定
- 顾问能理解价值点
- 后续值得继续投入

## 五、Stage 2：Demo v0.2 / Pre-MVP

### 阶段定位
主闭环已经成立后的增强版 Demo

### 阶段目标
在不进入重平台化的前提下，提高系统的：

- 可用性
- 稳定性
- 回看能力
- 输入处理能力
- 演示真实感

### 本阶段优先增强方向

#### A. 客户记录增强
- 历史画像版本展示
- 历史跟进消息回看
- 基础搜索
- 编辑客户显示名
- 更自然的客户详情工作台

#### B. 文档管理增强
- 文档列表优化
- 文档删除
- 文档失败原因展示
- 文档状态更清晰

#### C. AI 输出增强
- 画像字段稳定性优化
- 跟进消息风格差异优化
- 问答 citation 展示优化
- Prompt 版本记录增强

#### D. 输入预处理增强（可选）
- 聊天截图 OCR 入口
- 录音转写入口
- 标准化输入预处理层

### 本阶段不建议过早做的事
- 管理端大后台
- 团队协作权限体系
- 自动化营销流程
- 复杂多租户
- CRM 漏斗系统

### 本阶段成功标准
- Demo 更接近真实顾问工作流
- 体验明显优于 v0.1
- 产品演示更自然
- 为 MVP 做好更稳定的基础

### 阶段目标

- 输出稳定性
- 顾问持续使用路径
- 数据回看能力
- 基础产品化能力

### 本阶段可能纳入的能力

#### A. 客户工作流增强
- 更完整的客户详情页

- 画像版本回看
- 跟进消息版本回看

#### B. 知识库增强
- 更多文件格式支持
- 更稳定的检索链路
- 文档管理增强
- 更强问答结果可读性

#### C. 产品化增强
- 基础登录体系
- 基础角色控制
- 更稳定的错误处理与日志
- 更清晰的空状态和反馈系统
- 基础测试覆盖

#### D. Prompt 与 AI 可维护性增强
- Prompt 版本记录
- AI 输出质量调优

## 七、Stage 4：V1 正式产品雏形

### 阶段定位
面向更正式试点和更长期产品方向的第一版产品雏形

### 阶段目标
从“单点 AI 工具”进一步演化为：

**围绕教育咨询顾问工作流的 AI 工作台产品。**

### 本阶段可能纳入的能力

#### A. 协作与组织能力
- 更明确的用户角色体系
- 团队成员管理
- 顾问与负责人视角区分
- 基础协作边界

#### B. 更强产品化能力
- 管理后台
- 配置能力
- 文档分类与管理增强
- 更完整的历史记录系统

#### C. 更强 AI 能力
- 更成熟的 RAG 方案
- Prompt 管理与版本策略
- 更强输入预处理能力
- 质量监控与反馈机制

#### D. 更商业化的能力
- 小范围试点部署支持
- 更清晰的演示数据与业务模板
- 面向试点客户的产品包装

### 本阶段需要更谨慎评估的方向
- 多租户
- CRM 深度整合
- 自动触达
- 工作流自动化
- 经营分析与 BI
- 企业级合规与审计

这些能力即使进入 V1，也不一定都应该一起做。

### 本阶段成功标准
- 产品已有较稳定的顾问工作流定位
- 真实用户连续使用价值更清晰
- 系统结构足以支持后续商业化探索

## 八、当前版本与后续版本的边界关系

### Demo v0.1 关注
- 主闭环能不能成立

### Demo v0.2 / Pre-MVP 关注
- 主闭环能不能更自然、更稳定

### MVP v1.0 关注
- 用户能不能连续使用

### V1 关注
- 产品能不能逐步成为真正的业务系统雏形

## 九、当前最重要的版本边界提醒

当前开发中，最容易犯的错误是：

### 错误 1
把 V1 的想法塞进 Demo

### 错误 2
把 MVP 的增强项当成当前必须项

### 错误 3
因为“未来迟早要做”，就提前现在一起做

当前应始终坚持：

**Demo 先验证核心价值，不提前重平台化。**

## 十、版本推进的判断标准

是否可以从一个阶段进入下一个阶段，建议用以下问题判断：

### 从 Demo v0.1 → Demo v0.2
- 主闭环是否已经跑通？
- 输出是否已经基本稳定？
- 演示时是否能被理解？

### 从 Demo v0.2 → MVP
- 用户是否已经能看到连续使用价值？
- 系统是否已不再只是“单次展示工具”？
- 是否已出现值得继续打磨的真实使用反馈？

### 从 MVP → V1
- 是否有真实试点需求？
- 是否需要更强组织化能力？
- 是否已经值得为平台能力投入更多工程成本？
