## 一、文档目标

定义教育咨询顾问助手 Demo 的核心数据结构，确保：

- 前后端对实体理解一致
- Prisma schema 有明确依据
- API 设计有稳定数据基础
- 客户记录模块可以承接 AI 输出结果
- 后续从 Demo 演进到 MVP / V1 时不需要整体重建数据关系

---

## 二、设计原则

### 1. 围绕核心闭环建模
当前数据模型只服务于以下闭环：

- 文档上传
- 知识问答
- 客户画像提取
- 跟进消息生成
- 客户记录回看

### 2. 先轻量，再扩展
当前不做大而全 CRM 数据模型，只保留最必要实体。

### 3. AI 输出必须可持久化
客户画像和跟进消息不能只是临时结果，必须能保存、回看、再次使用。

### 4. 核心字段拆出，完整结果保留
对于画像这种 AI 输出，既要拆出高频字段，方便列表展示和筛选；也要保留完整结构化 JSON，方便后续扩展。

### 5. 预留未来升级空间
虽然当前不实现多租户、复杂权限、运营分析，但命名和关系设计不应阻断未来升级。

---

## 三、核心实体总览

当前建议的核心实体如下：

1. `users`
2. `knowledge_documents`
3. `document_chunks`
4. `clients`
5. `conversation_records`
6. `client_profiles`
7. `generated_followups`
8. `qa_logs`

其中：

- `knowledge_documents` 和 `document_chunks` 用于知识库问答
- `clients`、`conversation_records`、`client_profiles`、`generated_followups` 用于客户工作流
- `qa_logs` 用于问答留痕与后续质量回看
- `users` 用于预留系统用户概念

## 四、实体定义

---

## 1. users

### 作用
表示系统使用者，例如顾问或管理员。

### 当前定位
Demo 阶段可非常轻量，即使暂时不做完整登录，也建议先保留此实体概念。

### 建议字段
- `id`
- `email`
- `name`
- `role`
- `created_at`
- `updated_at`

### 字段说明
- `role`：Demo 阶段可先简单支持：
  - `admin`
  - `consultant`

### 当前用途
- 记录文档上传者
- 记录聊天记录创建者
- 记录跟进消息生成者
- 为未来权限扩展预留基础

---

## 2. knowledge_documents

### 作用
存储上传的知识库文档元数据。

### 建议字段
- `id`
- `title`
- `file_name`
- `file_type`
- `storage_key`
- `status`
- `raw_text`
- `processing_error`（可选，解析/入库失败时的可读说明）
- `deleted_at`（可选，软删除时间戳；非空则不出现在列表与检索）
- `uploaded_by`
- `created_at`
- `updated_at`

### 字段说明
- `title`：文档标题，可由文件名推导或手动定义
- `file_name`：原始文件名
- `file_type`：如 `pdf`、`txt`、`docx`
- `storage_key`：文件在对象存储中的路径
- `status`：文档处理状态，建议值：
  - `uploaded`
  - `processing`
  - `ready`
  - `failed`
- `raw_text`：提取后的纯文本，是否完整保留可视实际情况决定
- `processing_error`：失败状态时供文档管理与走查展示
- `deleted_at`：Demo 治理用软删除，不物理删文件/chunks（后续可接清理任务）
- `uploaded_by`：关联 `users.id`

### 当前用途
- 文档列表展示
- 文档状态展示
- 支撑问答检索链路
- 后续文档管理能力的基础

---

## 3. document_chunks

### 作用
存储文档切分后的片段，用于语义检索。

### 建议字段
- `id`
- `document_id`
- `chunk_index`
- `content`
- `embedding`
- `token_count`
- `metadata`
- `created_at`

### 字段说明
- `document_id`：关联 `knowledge_documents.id`
- `chunk_index`：chunk 在原文档中的顺序
- `content`：chunk 文本内容
- `embedding`：向量表示，建议与 pgvector 对应
- `token_count`：可选，用于调试或后续优化
- `metadata`：可选 JSON，记录页码、标题、段落信息等

### 当前用途
- 支撑问答检索
- 支撑 citations/snippet 返回
- 为未来检索优化保留空间

---

## 4. clients

### 作用
表示一个客户主体，是客户记录模块的核心承载对象。

### 为什么必须有这个实体
如果没有 `clients`，那么每次聊天记录提取出来的画像都会是散的，无法形成“客户记录列表”和“客户详情页”。

### 建议字段
- `id`
- `display_name`
- `student_name`
- `student_stage`
- `target_country`
- `budget_range`
- `current_stage`
- `source_note`
- `owner_user_id`
- `created_at`
- `updated_at`

### 字段说明
- `display_name`：顾问视角下的客户显示名，例如“张女士”。**全库唯一**（数据库唯一约束 + 写入前校验）；人工输入须满足最短长度（当前实现为 **≥ 3** 个字符）。新建客户时若顾问不填，由服务端生成**唯一占位名**。
- `student_name`：可选，学生姓名
- `student_stage`：例如 `G8`、`高一`、`本科申请`
- `target_country`：当前主要目标国家
- `budget_range`：预算范围，保留文本即可
- `current_stage`：当前业务阶段，建议轻量枚举：
  - `new_lead`
  - `initial_consultation`
  - `in_followup`
  - `high_intent`
  - `uncertain`
  - `closed`
- `source_note`：来源备注，可选
- `owner_user_id`：负责人，Demo 阶段可选

### 当前用途
- 客户记录列表的主表
- 画像、聊天记录、跟进消息的关联中心
- 后续客户详情页的基础

### 当前边界
这个实体不是完整 CRM 客户档案，只做轻量承载，不扩展到销售漏斗系统。

### 迁移说明（与实现对齐）
- 为 `display_name` 增加唯一约束时，若历史数据存在重复，迁移脚本需先**去重**再创建唯一索引（仓库中对应迁移如 `0004_client_displayname_unique`）。

## 5. conversation_records

### 作用
保存顾问输入的原始聊天记录。

### 建议字段
- `id`
- `client_id`
- `raw_text`
- `source_type`
- `created_by`
- `created_at`

### 字段说明
- `client_id`：关联 `clients.id`
- `raw_text`：完整聊天文本
- `source_type`：建议轻量枚举：
  - `manual_paste`
  - `imported`
- `created_by`：关联 `users.id`

### 当前用途
- 保存顾问输入的咨询对话
- 为画像提取提供输入依据
- 为跟进消息生成提供上下文来源
- 支撑客户详情页回看

### 当前关系
一个客户可以有多条聊天记录。

## 6. client_profiles

### 作用
保存从聊天记录中提取出的结构化客户画像。

### 建议字段
- `id`
- `client_id`
- `conversation_record_id`
- `student_stage`
- `target_country`
- `target_program`
- `budget_range`
- `timeline`
- `english_level`
- `parent_goals`
- `main_concerns`
- `risk_flags`
- `current_stage`
- `structured_json`
- `model_name`
- `prompt_version`
- `created_at`

### 字段说明
- `client_id`：关联 `clients.id`
- `conversation_record_id`：关联 `conversation_records.id`
- `student_stage`：学生阶段
- `target_country`：目标国家
- `target_program`：可选，目标项目/学校方向
- `budget_range`：预算范围
- `timeline`：时间线
- `english_level`：语言情况，可选
- `parent_goals`：家长/客户目标，可用文本或数组形式
- `main_concerns`：主要关注点
- `risk_flags`：风险点
- `current_stage`：当前成交阶段判断
- `structured_json`：完整结构化画像原始结果
- `model_name`：本次使用的模型
- `prompt_version`：本次使用的 prompt 版本

### 当前用途
- 客户详情页展示核心信息
- 跟进消息生成的基础输入
- 未来做画像版本对比的基础

### 设计说明
建议同时保留两类数据：

1. 常用字段拆出，用于列表和过滤
2. `structured_json` 保留完整结果，用于灵活扩展

这样既方便业务使用，也不损失 AI 结果的完整性。

## 7. generated_followups

### 作用
保存 AI 生成的跟进消息草稿。

### 建议字段
- `id`
- `client_id`
- `profile_id`
- `conversation_record_id`
- `style_type`
- `content`
- `model_name`
- `prompt_version`
- `created_by`
- `created_at`

### 字段说明
- `client_id`：关联 `clients.id`
- `profile_id`：关联 `client_profiles.id`
- `conversation_record_id`：关联 `conversation_records.id`
- `style_type`：建议枚举：
  - `wechat_short`
  - `semi_formal`
  - `english_optional`
- `content`：生成的跟进消息正文
- `model_name`：使用的模型名称
- `prompt_version`：使用的 prompt 版本
- `created_by`：关联 `users.id`

### 当前用途
- 展示跟进消息生成结果
- 允许客户详情页回看不同版本

### 当前关系

同一个画像也可以对应多种风格版本。

## 五、核心实体关系

### users 与 knowledge_documents
- 一个 user 可以上传多个 knowledge document

### knowledge_documents 与 document_chunks
- 一个 knowledge document 对应多个 document chunk

### clients 与 conversation_records
- 一个 client 可以有多条 conversation record

### clients 与 client_profiles
- 一个 client 可以有多条 profile 记录

### clients 与 generated_followups
- 一个 client 可以有多条 follow-up 记录

### conversation_records 与 client_profiles
- 一条 conversation record 通常生成一条主要 profile
- 后续若支持重跑，也可对应多条 profile

### client_profiles 与 generated_followups
- 一个 profile 可生成多条不同风格的 follow-up

### users 与 conversation_records / generated_followups / qa_logs
- 一个 user 可以创建多条聊天记录、跟进消息和问答日志

## 七、最小建模思路（关系视图）

可以按下面的关系理解：

- `User`
  - 拥有多个 `KnowledgeDocument`
  - 拥有多个 `ConversationRecord`
  - 拥有多个 `GeneratedFollowup`
  - 拥有多个 `QALog`

- `KnowledgeDocument`
  - 拥有多个 `DocumentChunk`

- `Client`
  - 拥有多个 `ConversationRecord`
  - 拥有多个 `ClientProfile`
  - 拥有多个 `GeneratedFollowup`

- `ConversationRecord`
  - 属于一个 `Client`
  - 可关联一个或多个 `ClientProfile`

- `ClientProfile`
  - 属于一个 `Client`
  - 来源于一条 `ConversationRecord`
  - 可关联多个 `GeneratedFollowup`

- `GeneratedFollowup`
  - 属于一个 `Client`
  - 可关联一个 `ClientProfile`
  - 可关联一条 `ConversationRecord`

## 九、当前刻意简化的点

为了控制 Demo 复杂度，当前有意识地不做以下设计：

### 1. 不做 organizations
当前默认单团队，不做组织表和多租户隔离。

### 2. 不做客户标签系统
不扩展 `client_tags`、标签筛选、标签运营。

### 3. 不做负责人协作流
虽然预留 `owner_user_id`，但不实现复杂分配与协作逻辑。

### 4. 不做活动时间轴系统
不额外建设 `activity_logs` 作为完整 CRM 操作流。

### 5. 不做审核状态流
不增加 `approved`、`reviewed`、`audit_status` 等复杂状态。

### 6. 不做 Prompt 配置平台表
Prompt 版本先用文本字段记录，不单独设计复杂配置表。

## 十一、当前建模重点提醒

### 1. clients 是客户记录模块的中心
你新增的“客户记录模块”能否成立，关键在于 `clients` 是否成为统一承载对象。

### 2. 画像不是覆盖式单字段，而应允许多版本记录
不要把客户画像直接只放在 `clients` 表里，否则后续无法保留历史提取结果。

### 3. followup 应单独成表
不要只把最新消息塞到客户表里，否则无法支持多种风格版本与历史回看。

### 4. 高频字段要拆出
例如 `student_stage`、`target_country`、`current_stage` 这些字段建议拆出，而不是全部塞进 JSON。

### 5. 完整结构仍要保留
AI 输出天生会迭代，所以 `structured_json` 非常重要。
