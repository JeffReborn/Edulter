## 二、架构设计原则

### 1. 先跑通闭环，再做复杂化
Demo 第一版的目标是稳定完成以下闭环：

1. 上传内部资料
2. 基于资料进行知识问答
3. 粘贴聊天记录
4. 提取客户画像
5. 生成跟进消息
6. 保存到客户记录中

当前不追求平台级复杂度。

### 2. 前后端统一技术栈
优先采用统一技术栈，降低维护复杂度，便于 Cursor 理解和持续开发。

### 3. AI 能力模块化
将以下能力拆开设计，避免互相耦合：

- 文档处理
- 知识问答
- 客户画像提取
- 跟进消息生成
- 客户记录查看

### 4. AI 输出必须可持久化
画像和跟进消息不是一次性结果，而是后续工作流的一部分，必须可保存、可回看、可再次调用。

### 5. 为后续升级预留空间
虽然当前只做 Demo，但架构应允许后续增加：

- 历史记录增强
- 文档管理增强
- 权限系统
- 多租户
- Prompt 版本管理
- 管理后台

- Next.js 15
- React
- TypeScript
- Tailwind CSS

### 后端
- Next.js Route Handlers

- Demo 阶段可先保留轻量方案
- 如暂不做正式登录，也建议在数据模型中预留 user 概念

## 四、系统总体分层

整个系统建议拆成 6 层。

### 1. 展示层（Frontend UI）

负责：
- 页面展示
- 输入收集
- 结果显示
- loading / error / empty state

主要页面建议：

- 主工作台
- 文档上传 / 文档管理页
- 知识问答页
- 客户画像与跟进消息页
- 客户记录列表页
- 客户记录详情页

这一层不直接处理复杂业务逻辑，只负责交互和展示。

### 2. API 层（Application API）

负责：
- 接收前端请求
- 参数校验
- 调用后端服务
- 返回结构化响应

主要 API 模块建议：

- `/api/documents/*`
- `/api/qa/*`
- `/api/profiles/*`
- `/api/followups/*`
- `/api/clients/*`

这一层的目标是：  
前端永远消费稳定的 JSON 结构，而不是直接消费模型原始输出。

### 3. 业务服务层（Domain Services）

负责具体业务逻辑封装。

建议拆分为以下服务：

- `documentService`
- `retrievalService`
- `qaService`
- `profileService`
- `followupService`
- `clientService`

职责示例：

#### `documentService`
负责：
- 文件上传后存储
- 文本提取
- 文档状态维护

#### `retrievalService`
负责：
- chunk 检索
- retrieval 逻辑
- 返回相关上下文片段

#### `qaService`
负责：
- 组织问答 Prompt
- 调用模型回答
- 组装 citations

#### `profileService`
负责：
- 从聊天记录提取客户画像
- 输出结构化 JSON
- 写入画像记录

#### `followupService`
负责：
- 基于画像和聊天记录生成跟进消息
- 输出多版本文本

#### `clientService`
负责：
- 创建客户主体
- 关联聊天记录、画像、跟进消息
- 查询客户列表与详情

### 4. AI 服务层（AI Layer）

负责所有模型调用与输出控制。

建议包含：

- `embeddingService`
- `llmService`
- `promptBuilder`
- `outputParser`

职责如下：

#### `embeddingService`
- 生成文档 chunk embedding
- 支持 retrieval 所需的向量化处理

#### `llmService`
- 统一封装模型调用
- 隐藏底层供应商差异

#### `promptBuilder`
- 构造问答、画像提取、跟进消息生成 Prompt
- 保持 Prompt 逻辑集中管理

#### `outputParser`
- 校验模型输出格式
- 将自由文本整理成可落库、可展示的结构化数据

这一层很重要，因为你的产品本质上是“业务系统 + AI 能力层”，而不是纯前后端 CRUD。

### 5. 数据持久化层（Persistence Layer）

负责保存和读取核心业务数据，包括：

- 上传的知识库文档
- 文档切分后的 chunks
- 客户主体记录
- 聊天记录
- 客户画像
- 跟进消息
- 问答日志（可选）

这一层的重点不是“数据量很大”，而是“AI 输出结果必须可回看、可追踪、可关联”。

### 6. 基础设施层（Infrastructure）

负责系统运行所依赖的底层能力：

- 数据库连接
- 文件存储
- OpenAI API 调用
- 环境变量配置
- 日志
- 基础错误处理

后续若进入正式产品阶段，这一层还可继续扩展：

- 监控
- 队列
- 任务调度
- 审计日志
- 配置中心

## 五、核心模块架构关系

### 模块 1：文档上传模块

输入：
- 内部资料文件

处理流程：
1. 前端上传文件
2. 后端接收文件
3. 文件保存到对象存储
4. 提取纯文本
5. 切分为 chunks
6. 生成 embedding
7. 写入文档表与 chunk 表
8. 更新文档状态

输出：
- 文档可被知识问答使用
- 文档状态可在前端查看

关联文档：
- [[文档上传模块]]

### 模块 2：知识问答模块

输入：
- 顾问输入的问题

处理流程：
1. 接收问题
2. 检索相关 chunk
3. 构造问答 Prompt
4. 调用 LLM
5. 返回 answer + citations + source info

输出：
- 标准答案
- 至少一条出处
- 来源文档信息

关联文档：
- [[知识问答模块]]

### 模块 3：客户画像提取模块

输入：
- 顾问与客户聊天记录
- 可选客户名称备注

处理流程：
1. 接收聊天记录
2. 调用画像提取 Prompt
3. 输出结构化画像 JSON
4. 创建或更新客户主体
5. 保存聊天记录
6. 保存客户画像

输出：
- 结构化客户画像
- 可关联到客户记录

关联文档：
- [[客户画像提取模块]]

### 模块 4：跟进消息生成模块

输入：
- 客户 ID 或客户画像
- 最近聊天记录
- 风格类型

处理流程：
1. 读取客户画像
2. 读取最近聊天记录
3. 可选读取知识库中的标准表述
4. 构造跟进消息 Prompt
5. 调用 LLM
6. 生成多个版本消息
7. 写入生成记录

输出：
- 可编辑的跟进消息草稿

关联文档：
- [[跟进消息生成模块]]

### 模块 5：客户记录模块

输入：
- 已生成的客户画像
- 原始聊天记录
- 跟进消息生成结果

处理流程：
1. 展示客户记录列表
2. 根据客户 ID 查看详情
3. 展示最新聊天记录
4. 展示最新画像
5. 展示最近生成的跟进消息
6. 允许再次触发提取或生成

输出：
- 顾问可持续回看和继续跟进的客户记录

说明：
这个模块是“AI 结果的持久化容器”，不是完整 CRM。

关联文档：
- [[客户记录模块]]

## 六、核心数据流

### 闭环 A：知识问答

1. 上传内部资料
2. 文档被处理并入库
3. 用户输入问题
4. 系统检索相关内容
5. 模型基于相关内容回答
6. 返回答案与出处

### 闭环 B：聊天记录 → 画像 → 跟进 → 客户记录

1. 顾问粘贴聊天记录
2. 系统提取客户画像
3. 创建或更新客户主体
4. 保存聊天记录与画像结果
5. 顾问触发跟进消息生成
6. 系统生成多版本消息
7. 保存生成结果
8. 顾问在客户记录中回看和继续操作

## 九、当前架构风险点

### 1. 文档解析质量不稳定
不同文件类型质量差异较大，首版应优先支持少数高价值格式。

### 2. RAG 效果不稳定
Demo 阶段不做高级检索优化，但必须保证“有出处链路”。

### 3. 模型输出不稳定
客户画像和跟进消息必须强约束 schema，不能直接依赖自由输出。

### 4. 客户记录模块容易滑向 CRM
必须始终坚持其定位：
- 保存
- 查看
- 回看
- 继续生成

而不是：
- 管理商机
- 分配负责人
- 漏斗分析
- 自动提醒

### 5. 开发优先级容易失衡
先做 AI 主闭环，再补客户记录列表与详情页，不要一开始先做一堆管理 UI。
